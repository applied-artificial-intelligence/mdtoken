## Recent Decisions & Technical Context

### Tokenizer Configuration Design

**Decision**: Support both `encoding` (power users) and `model` (convenience)
- `encoding`: Direct tiktoken encoding name (e.g., "cl100k_base", "o200k_base")
- `model`: User-friendly model names mapped to encodings (e.g., "gpt-4" → "cl100k_base")

**Rationale**:
- Power users want direct encoding control
- Most users prefer simple model names
- Encoding takes precedence if both specified (expert override)
- Clear error messages with available options when model not recognized

**Implementation Pattern**:
```python
def __init__(self, ..., encoding=None, model=None):
    if encoding is not None:
        self.encoding = encoding
    elif model is not None:
        self.encoding = self._resolve_model_encoding(model)
    else:
        self.encoding = self.DEFAULT_CONFIG["encoding"]  # cl100k_base
```

### Configuration System

**Loading Hierarchy**:
1. Check for explicit config path via `--config` flag
2. Look for `.mdtokenrc.yaml` in current directory
3. Fall back to `DEFAULT_CONFIG`

**Pattern Matching**:
```python
def get_limit(self, file_path: str) -> int:
    # 1. Exact path match (highest priority)
    if file_path in self.limits:
        return self.limits[file_path]

    # 2. Pattern match (substring or endswith)
    for pattern, limit in self.limits.items():
        if pattern in file_path or file_path.endswith(pattern):
            return limit

    # 3. Default (fallback)
    return self.default_limit
```

**Validation Strategy**:
- Validate on `__init__` (fail fast)
- Clear error messages with context
- Type checking for all parameters
- Helpful suggestions when validation fails
